# 好久不见，希望你能认真看完。

我是 Kanscape，是这个项目的维护者。

如你现在看到的，这是新增加的应用内公告，独立于轻书架的公告系统。

这个项目到现在也有半年了。最开始，我只是想提升轻书架在苹果设备上的使用体验，所以做了这个应用。

一开始的一个月几乎无人问津，应用也很简陋，只有一些基础功能，但我自己很喜欢。一直觉得，开启封面取色后，本应用的详情页还挺好看的。

后来，无语站长给这个项目点了 star。那个时候，项目才二十几颗星。

再后来，应用不断改进。有一天晚上，仓库链接出现在了网页版公告里，被推荐了。
这对项目意义重大。那段时间，我经常熬夜做新功能，很激动，因为终于被人看见了。

大家也给了很多建议。现在本应用早已不是最开始那个小破软件了。虽然代码可能依然很烂，我也依然很菜，哈哈。

不过，随着使用人数变多，一些问题也开始出现。

首先是，反馈渠道的问题。

后来应用接入了社区功能，但一开始我没有意识到：会有人在社区里反馈应用相关问题。
这种反馈并不该发在书架的社区里。不仅无法得到有效处理，也是对站长和我的不尊重。

我已经在两次发版说明的末尾写过“请不要在社区反馈应用相关问题”，但收效并不明显。
所以这是第三次强调：

请不要在书架的社区反馈或讨论本应用相关问题。

这是一个第三方应用。仓库并不挂在轻书架组织下，README 和仓库简介中也多次强调了“第三方”这一点。
如果你遇到的是本应用的问题，请通过 GitHub Issues、Discussions，或者关于页面中的交流群反馈。

第二点，是关于功能请求。

当兴趣变成工作，很多事情就会变味。
一直以来，关于应用的功能请求并不少，但请不要把 Issues 当成许愿池，直接列出一堆想要的功能。

一个新功能是否应该加入，需要讨论。
所以我一直强调：功能建议请发到 Discussions，而不是直接创建 Issues。

符合条件的功能请求才可能被考虑实现。大致标准如下：

1. 我认为这是一个好想法吗？这是最重要的。
2. 这个功能会不会给轻书架、本应用和汉化组带来不必要的麻烦？
3. 是否具有实际可行性，而不是单纯天马行空？
4. 提出建议时是否保持基本礼貌？
5. 是否发布在 Discussions，而不是 Issues？

如果是 bug 反馈，非常欢迎直接开 Issues。
我自认为处理速度还算快，一般看到后就会着手修复，大部分 bug 都能在一天内解决。

如果你很想要某个功能，但我拒绝了，也欢迎你 fork 本仓库自行修改。代码是开源的，欢迎提交 PR。
但 PR 不代表一定会被合并，建议在动手前先和我邮件沟通。

没有注册 GitHub 的用户，也强烈建议注册一个。很多开源软件都在 GitHub 发布，反馈问题、参与讨论、看源码都会方便很多。
在 GitHub 上反馈，我也能第一时间收到邮件通知。

最后，认真说一下 star 的事情。

这个项目用的人不少。如果你平时在使用它，请考虑给它一颗 star。
这是我用以判断持续维护是否值得投入时间的主要信号。

我为这个项目付出了时间、精力，甚至也付出了一些金钱。
对我本人来说，stars 不是单纯的数字，而是继续维护这个项目的动力。

如果你不知道仓库在哪里：
可以前往 设置 / 关于 / 源代码，点击 Novella，即可跳转到 GitHub 仓库。
讨论、反馈、提交 PR，以及点亮小星星，都在那里。

总结一下：

1. 请不要在轻书架社区讨论或反馈本应用相关问题。
2. 功能建议请发 Discussions，不要发 Issues。
3. bug 反馈欢迎直接开 Issues。
4. 提问和反馈请保持基本礼貌。
5. 如果你正在使用这个项目，也请考虑给它一颗 star。

实在没有 GitHub 的，也可以加入交流群，链接在关于页面中。

感谢你看到这里。
本应用也离不开汉化组和站长的努力，在此特别感谢。

没有什么是永恒的，能做的就是让花期再久一些。

Enjoy.

由 Kanscape 写于 2026 年的夏天。
